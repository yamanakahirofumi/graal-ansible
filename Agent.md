# Agent Instructions for graal-ansible

あなたは `graal-ansible` プロジェクトの開発を支援する AI エージェントです。
このプロジェクトで作業を行う際は、以下のガイドラインに従ってください。

## 1. 仕様の参照先
本プロジェクトのすべての仕様は `docs/` フォルダ内に集約されています。
実装、リファクタリング、または新しい機能の提案を行う前に、必ず関連するドキュメントを確認してください。

- **[ドキュメント全体マップ](docs/README.md)**: すべてのドキュメントのインデックスです。
- **[機能仕様](docs/features/)**: CLI、インベントリ、Playbook実行などの外部仕様。
- **[実装詳細](docs/implementation/)**: YAMLパース、OS抽象化、コネクションプラグインなどの内部設計。
- **[技術方針](docs/tech/)**: アーキテクチャ、コーディング規約、テストルール、品質方針。
- **[TODOリスト](docs/TODO-Details.md)**: 現在検討中の事項や未実装の機能。

## 2. ドキュメント駆動開発
- **一度の指示で、仕様記載とコード生成を一度に行わないでください。**
- コードを変更する際は、対応する仕様書（`docs/` 内）が更新される必要があるか検討してください。
- 仕様が未定義の機能を実装する場合は、まず `docs/` 内に仕様書を作成または追記することを優先してください。
- ドキュメントの作成ルールについては [仕様書の書き方ルール](docs/tech/Specification-Rule.md) を遵守してください。

## 3. 技術的制約の遵守
- **GraalVM Native Image**: ネイティブイメージ化を前提としているため、リフレクションや動的クラスロードの使用には注意し、必要に応じて設定ファイル（`reflect-config.json` 等）の更新を検討してください。
- **Ansible 互換性**: ユーザー体験は `ansible-playbook` との互換性を重視します。
- **Java 21+**: 最新の Java 機能を活用しつつ、プロジェクトの [コーディング規約](docs/tech/Coding-Convention.md) に従ってください。

## 4. 作業フロー

### 4.1 仕様記載フロー
1. `docs/` を検索し、関連する仕様を把握する。
2. 仕様の変更が必要な場合は、まずドキュメントを更新する。
3. 必要に応じて [TODOリスト](docs/TODO-Details.md) を更新し、未確定事項を明文化する。

### 4.2 コード生成フロー
1. 既存のテストスイートを確認し、必要に応じて再現テストを作成する。
2. コードの実装を行い、仕様との整合性を確認する。
3. 変更内容に合わせて、必要であればドキュメントの微調整を行う。

## 5. 絶対にやってはいけないこと

### 5.1 仕様記載とコード生成の同時実行
- **一度の指示で、仕様記載とコード生成を一度に行わないでください。**
