# TODOリストの書き方ルール

本プロジェクトにおいて、機能追加や技術的な課題を管理するための「TODOリスト（検討事項）」の記述ルールを定義します。

## 1. 管理場所
- メインの管理ファイル: `docs/TODO-Details.md`
- 関連ドキュメントからの参照: `docs/README.md` 等から上記の詳細ファイルへリンクを貼る。
- **原則**: `TODO-Details.md` には、**設計面（アーキテクチャ、アルゴリズム、データ構造、インターフェース定義、技術的課題の解決方針など）のことのみ**を記載してください。単純な実装（コーディング）タスクやバグ修正は含めません。

## 2. 記述項目と構成

TODO項目は以下の構成で記述します。

### 2.1 未完了の項目
- **チェックボックス**: `[ ]` を使用。
- **タイトル**: 検討が必要な機能や課題の名称。
- **概要**: なぜその検討が必要なのか、解決すべき問題の要約。
- **検討内容**: 具体的にどのような点を決める必要があるか、または調査すべき事項を箇条書きで記載。

例：
```markdown
### [ ] インベントリ管理の基本設計
- **概要**: ターゲットホストの静的ファイル（INI/YAML）からの読み込み。
- **検討内容**:
    - 静的ホストグループのサポート範囲。
    - 階層化されたグループ変数の優先順位解決ルール。
```

### 2.2 完了済みの項目
- **チェックボックス**: `[x]` に更新。
- **完了日**: 解決または仕様策定した日付（YYYY-MM-DD形式）を記載。
- **解決策**: 仕様が策定された場合、その仕様書（Markdownファイル）へのリンクを明記する。

例：
```markdown
### [x] 実行エンジンの基本設計
- **完了日**: 2026-02-22
- **概要**: 複数タスクの順次実行とエラーハンドリングの基本方針。
- **解決策**: [タスク実行エンジンの設計](../implementation/Task-Executor.md) にて、〇〇パターンの採用を策定済み。
```

## 3. カテゴリ分け
TODOリストは以下のカテゴリに分類して管理します。
1. **技術面**: パフォーマンス、マルチスレッド、データ構造、アーキテクチャ設計。
2. **機能面**: Playbookの解釈、インベントリ管理、モジュール呼び出し。
3. **実装時の詳細事項**: 開発中に見つかった、より踏み込んだ技術的・設計的課題。
4. **人間が主導すべき設計・調整面**: 外部仕様との互換性ポリシー、コアアーキテクチャの意思決定。

## 4. 人間が考えるべき事項の取り扱い
TODOリストには、AIが効率的に生成可能な「定型的な実装」ではなく、人間がプロジェクトの方向性や一貫性を基準に判断・設計すべき項目を含めます。これらの項目には、検討内容に「人間が考えるべきこと」であることを明記してください。

- **AIが得意なこと**: 定型的なコードの生成、既存パターンの拡張、ドキュメントの整理、単体テストの雛形作成。
- **人間が主導すべきこと**: 全体的なアーキテクチャ設計、Ansibleとの互換性維持に関する重要判断、未踏の技術領域へのアプローチ。

## 5. 更新のタイミング
- 新しい機能の構想が浮上したが、詳細な仕様が未定の時。
- 実装中に、現在の仕様では考慮不足な点（エッジケース等）を発見した時。
- 仕様が確定し、詳細ドキュメントが作成された時（ステータスを完了に更新）。
