# 仕様書の書き方ルール

本プロジェクトにおけるドキュメント作成のガイドラインです。

## 1. ドキュメント構成

ドキュメントは Markdown 形式 (`.md`) で記述し、以下のディレクトリ構造に従って配置します。

- **`docs/features/`**: 機能仕様（ユーザー視点の仕様）。
- **`docs/tech/`**: 技術設計、コーディング規約、CI/CD 設定。
- **`docs/implementation/`**: 内部実装の詳細、ライブラリの利用方法。

## 2. 記述の原則

- **明確さと簡潔さ**: 一文を短く保ち、要点を箇条書きで示す。
- **図解の活用**: 複雑なフローや構造は、Mermaid 等の図解ツールを使用して視覚化する（将来的な対応を検討）。
- **用語の統一**: プロジェクト内で使用する用語（Playbook, Task, Host, Inventory 等）は Ansible 本家の定義に準拠する。

## 3. ヘッダーとセクション

- `#` (H1): ファイル名に対応する大見出しを 1 つだけ記述。
- `##` (H2): 主要なセクション。
- `###` (H3): 詳細な項目。

## 4. テーブルの活用

比較や設定値、オプションの一覧などは Markdown テーブルを活用して見やすく整理します。

## 5. リンクの記述

- 関連するドキュメントへのリンクには、相対パスを使用します。
    - 例: `[技術スタック](../tech/Tech-Stack.md)`

## 6. 更新とレビュー

- コードの変更に伴い仕様が変わる場合は、必ず対応するドキュメントも同時に更新します。
- 新機能の追加時は、必ず `docs/features/` に仕様書を作成してください。
