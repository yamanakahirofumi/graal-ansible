# 実際のコレクションを用いた自動テストの要件

本ドキュメントは、GraalPy 環境で実際の Ansible コレクション（`ansible.builtin` 等）を呼び出し、その動作を検証するための自動テストに必要な要件をまとめたものです。

## 1. 実行環境の要件

### 1.1 GraalPy のセットアップ
- テスト実行時に GraalPy コンテキストが正しく初期化されていること。
- Java から Python モジュールを呼び出すための Polyglot API が利用可能であること。

### 1.2 Ansible コレクションの配置と自動取得
- テストに使用するコレクション（例：`ansible.posix`, `community.general` 等）が、テスト実行環境の特定のパス（`ANSIBLE_COLLECTIONS_PATH`）に配置されていること。
- **自動取得の検討**: テスト実行前に `ansible-galaxy collection install` 相当の処理（現時点でも標準的なコマンド）を自動で行い、必要なコレクションを指定した一時ディレクトリ等に配備する仕組みを検討すること。
- `ansible.builtin` モジュールについては、Ansible 本体（あるいは互換レイヤー）からロード可能であること。
- **ツール選定**: 現行の Ansible 環境では `ansible-galaxy` が依然としてコレクション管理のデファクトスタンダードであり、自動テスト時もこのコマンド、または `requirements.yml` を用いたインストール手法を採用することが適切である。

### 1.3 Python 依存ライブラリの解決
- 各コレクションが依存する Python ライブラリ（例：`requests`, `pyyaml` 等）が、テスト実行時の Python パスに含まれていること。
- 必要に応じて、`pip` を用いた動的なインストール、またはプリインストールされた環境の利用を検討すること。

## 2. テスト実装の要件

### 2.1 入力データの整合性
- Java 側で定義したタスク引数（Map/Record）が、Python 側のモジュールが期待する JSON 形式に正しく変換されること。
- complex_args（複雑な構造の引数）の受け渡しがサポートされていること。

### 2.2 実行結果の取得とパース
- モジュールの標準出力（JSON）をキャッチし、Java の `TaskResult` オブジェクトに正しくマッピングできること。
- `changed`, `failed`, `msg` などの標準的なフィールドが欠落なく取得できること。

### 2.3 副作用の検証
- `ansible.builtin.copy` 等のファイル操作を伴うモジュールの場合、実際にファイルが作成・変更されたかを Java のファイル IO API を用いて検証できること。
- 冪等性（2回実行しても 2回目は `changed=false` になること）の検証が可能であること。

## 3. テスト分離と独立性の要件

### 3.1 ファイルシステムの分離
- テストによってホスト環境が汚染されないよう、一時ディレクトリ（`JUnit @TempDir` 等）を作業ディレクトリとして使用すること。
- テスト終了後に作成されたファイルやディレクトリが自動的に削除されること。

### 3.2 実行状態のクリーンアップ
- 各テストケース間で Python インタプリタの状態（グローバル変数等）が干渉しないよう、可能な限り独立したコンテキストで実行すること。

## 4. パフォーマンスとデバッグの要件

### 4.1 実行時間の許容範囲
- 実際の Python モジュールを起動するため、モックテストに比べて実行時間が長くなることを許容しつつ、CI 上で現実的な時間内に終了すること。

### 4.2 ログとエラー情報の出力
- テスト失敗時に、Python 側のスタックトレースや標準エラー出力が Java のテストログに詳細に出力されること。
