# タスク制御の実装詳細 (Task Control Implementation)

本ドキュメントでは、`graal-ansible` におけるタスクの実行制御（条件分岐、ループ、変数登録、ハンドラー、ブロック、リトライ等）の実装方法について詳述します。

## 1. 条件分岐 (`when`)

タスクの実行を特定の条件に基づいて制御します。

- **実装方針**:
    - `when` 句に記述された Jinja2 式を `VariableResolver` (Jinjava) を用いて評価します。
    - 評価結果が `true` または真とみなされる値（非空の文字列、非ゼロの数値等）の場合のみ、タスクを実行します。
    - 評価結果が `false` の場合、そのホストでのタスク実行はスキップされ、結果には `skipped=true` が含まれます。
    - `when` 句がリスト形式で記述されている場合は、すべての条件が `true` であること（論理積）を確認します。
- **留意点**:
    - Jinja2 の式として評価するため、変数は `{{ }}` で囲まずに記述されることが一般的であることに対応します。
- **データ構造**:
    - `Task` レコードに `Object when` フィールド（String または List<String>）を保持します。

## 2. ループ (`loop`)

同一のタスクを複数のアイテムに対して繰り返し実行します。

- **実装方針**:
    - `loop` フィールドに指定されたリスト（またはリストを返すテンプレート）を解析します。
    - リストの各アイテムについて、特殊変数 `item` にその値を割り当て、タスクを繰り返し実行します。
    - 各イテレーションの結果は、最終的なタスク結果の `results` リストに格納されます。
- **レガシー対応**:
    - `with_items` 等のレガシーなループ構文は、内部的に `loop` 処理へ変換することを検討します。
- **データ構造**:
    - `Task` レコードに `Object loop` フィールドを保持します。

## 3. 変数登録 (`register`)

タスクの実行結果を後のタスクで利用するために変数として保存します。

- **実装方針**:
    - タスク実行後、`register` キーで指定された名前で、実行結果（`TaskResult` のデータ部分）を `VariableManager` に登録します。
    - 登録された変数は、同一 Play 内の以降のタスクで参照可能です。
- **データ構造**:
    - `Task` レコードに `String register` フィールドを保持します。
    - 登録される変数は Map 形式とし、Ansible 互換のフィールド名（`rc`, `stdout`, `stderr`, `changed`, `failed` 等）を保持します。

## 4. ハンドラーと通知 (`handlers`, `notify`)

タスクによって状態が変更された（`changed=true`）場合にのみ実行される特別なタスクを制御します。

- **実装方針**:
    - **通知 (Notify)**: タスクが実行され、結果が `changed=true` かつ `notify` 句がある場合、指定されたハンドラー名を「実行待ちキュー」に登録します。
    - **実行タイミング**: プレイ（Play）内のすべての通常タスクが完了した直後、または `flush_handlers` タスクが実行されたタイミングで、キューに溜まったハンドラーを順次実行します。
    - **冪等性**: ハンドラーは、一つのプレイ内で何度通知されても、最後に一度だけ実行されます。
- **データ構造**:
    - `Task` レコードに `List<String> notifications` フィールドを保持します。
    - `Play` レコードに `List<Task> handlers` フィールドを保持します。

## 5. ブロック (`block`, `rescue`, `always`)

複数のタスクを論理的にグループ化し、エラーハンドリング（例外処理）を行います。

- **実装方針**:
    - **block**: グループ化された一連のタスクを順次実行します。
    - **rescue**: `block` 内のタスクでエラー（failed）が発生した場合にのみ実行されるタスク群です。
    - **always**: `block` の成否に関わらず、最後に必ず実行されるタスク群です。
- **留意点**:
    - ブロック自体も `when` 句による条件分岐が可能です。
- **データ構造**:
    - `Task` またはそれに準ずる `Block` レコードとして定義し、タスクのリストを保持します。

## 6. 結果のカスタマイズ (`failed_when`, `changed_when`)

モジュールの標準的な成功/変更判定を上書きします。

- **実装方針**:
    - **failed_when**: 指定された条件式が `true` の場合、モジュールが成功していても `failed=true` とみなします。
    - **changed_when**: 指定された条件式が `true` の場合、モジュールに変更がなくても `changed=true` とみなします。逆に `false` を指定して常に変更なしにすることも可能です。
- **実行タイミング**: モジュール実行完了直後、`register` や `notify` の処理前に行います。

## 7. リトライ処理 (`until`, `retries`, `delay`)

タスクが特定の条件を満たすまで、繰り返し実行します。

- **実装方針**:
    - **until**: 条件式が `true` になるまで繰り返します。
    - **retries**: 最大試行回数（デフォルト 3）。
    - **delay**: 試行間の待ち時間（秒、デフォルト 5）。
- **留意点**:
    - 各試行の結果は `register` された変数（`results` リスト）に蓄積される必要があります。

## 8. 委譲と実行制御 (`delegate_to`, `run_once`, `ignore_errors`)

- **delegate_to**: タスクを指定された別のホスト（例: localhost）で実行します。
- **delegate_facts**: `delegate_to` 使用時に、収集されたファクトを委譲先ではなく元のホストに割り当てるかを制御します。
- **run_once**: 複数のターゲットホストがある場合でも、1つのホストで1回だけ実行します。
- **ignore_errors**: タスクが失敗しても、そのホストでのプレイ実行を継続します。
- **ignore_unreachable**: ホストが到達不能（unreachable）な場合にエラーを無視します。

## 9. 実行エンジン (`PlaybookExecutor`) での処理順序

各タスクの実行時、以下の順序で制御ロジックを適用します。

1.  **委譲の判定**: `delegate_to` がある場合、接続先を切り替えます。
2.  **ループの展開**: `loop` がある場合、リストを取得しイテレーションを開始します。
3.  **条件評価**: `when` 句を評価し、実行の可否を判断します。
4.  **リトライ制御**: `until` がある場合、条件を満たすか最大回数に達するまで繰り返します。
5.  **タスク実行**: `TaskExecutor` を通じてモジュールを実行します。
6.  **結果のカスタマイズ**: `failed_when`, `changed_when` を評価し、結果を補正します。
7.  **結果登録**: `register` がある場合、結果を保存します。
8.  **通知**: `changed=true` かつ `notify` がある場合、ハンドラーをマークします。
9.  **エラーハンドリング**: 失敗時に `ignore_errors` が `false` ならばそのホストの実行を停止し、必要に応じて `rescue` ブロックへ遷移します。
