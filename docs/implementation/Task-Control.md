# タスク制御機能の実装詳細 (Task Control Implementation)

本ドキュメントでは、Ansible の主要なタスク制御機能である条件分岐 (`when`)、変数登録 (`register`)、ループ (`loop`)、およびハンドラー (`handlers`) の `graal-ansible` における実装詳細を定義します。

## 1. 条件分岐 (Conditional: `when`)

タスクの実行可否を動的に判断する仕組みです。

- **実装方法**:
    - `TaskExecutor` はタスク実行前に、`when` 句に記述された式を `VariableResolver` (Jinjava) を用いて評価します。
    - 評価結果が `true` の場合のみタスクを実行します。
    - `when` 句がリスト形式で記述されている場合は、すべての条件が `true` であること（論理積）を確認します。
- **留意点**:
    - Jinja2 の式として評価するため、変数は `{{ }}` で囲まずに記述されることが一般的であることに対応します。

## 2. 変数登録 (Registration: `register`)

タスクの実行結果を後続のタスクで利用可能にする仕組みです。

- **実装方法**:
    - `TaskResult` の内容（`changed`, `failed`, `stdout`, `json` 等）を、`register` で指定された変数名で `VariableManager` に格納します。
    - 変数のスコープは、そのホストの実行コンテキスト（Host Vars 相当）に準じます。
- **データ構造**:
    - 登録される変数は Map 形式とし、Ansible 互換のフィールド名（`rc`, `stdout`, `stderr`, `changed`, `failed` 等）を保持します。

## 3. ループ (Loops: `loop`)

単一のタスクをリストの各要素に対して繰り返し実行する仕組みです。

- **実装方法**:
    - `PlaybookExecutor` はタスクに `loop` 句が含まれている場合、そのリストを展開します。
    - 展開された各要素に対して、一時的な変数 `item` を割り当て、タスクを複数回実行する「タスクのリスト」として扱います。
    - 各イテレーションの結果は集約され、最終的な `TaskResult` に `results` リストとして格納されます。
- **レガシー対応**:
    - `with_items` 等のレガシーなループ構文は、内部的に `loop` 処理へ変換することを検討します。

## 4. ハンドラー (Handlers: `notify` & `handlers`)

タスクによって状態が変更された場合にのみ実行される特殊なタスクです。

- **実装方法**:
    - **通知 (Notify)**: タスクが実行され、結果が `changed=true` かつ `notify` 句がある場合、指定されたハンドラー名を「実行待ちキュー」に登録します。
    - **実行タイミング**: プレイ（Play）内のすべての通常タスクが完了した直後、または `flush_handlers` タスクが実行されたタイミングで、キューに溜まったハンドラーを順次実行します。
    - **冪等性**: ハンドラーは、一つのプレイ内で何度通知されても、最後に一度だけ実行されます。
- **データ管理**:
    - `PlaybookExecutor` がプレイごとの「通知済みハンドラーセット」を管理します。

## 5. 実行エンジンへの統合

これらの制御機能は、`PlaybookExecutor` および `TaskExecutor` の実行ループ内に以下のように組み込まれます。

1.  **ループ展開**: `loop` があればタスクを複製。
2.  **条件評価**: `when` を評価し、`false` ならスキップ。
3.  **タスク実行**: `TaskExecutor` によりモジュールを実行。
4.  **変数登録**: `register` があれば結果を保存。
5.  **ハンドラー通知**: `changed=true` なら `notify` 先をマーク。
