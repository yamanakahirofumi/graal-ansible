# 本物のコレクション実行に向けた段階的実装ロードマップ

本ドキュメントでは、本物の Ansible コレクションを GraalPy 上で実行可能にするための段階的な実装計画を定義します。

## 1. 背景と目的

[モジュールの開発方針](../features/Module-Development-Policy.md)に基づき、本プロジェクトでは本物の Ansible コレクションを使用することを原則としています。しかし、コレクションは Ansible 本体の内部ライブラリ（`ansible.module_utils` 等）に強く依存しており、GraalPy 上での完全な動作には多くの技術的ハードルがあることが判明しました。これらを解決し、最終的に安定かつ高速な実行環境を実現するために、以下の 3 段階のフェーズで実装を進めます。

## 2. フェーズ 1：Ansible 本体の完全ロードと基本動作の実現

このフェーズでは、まず「動くこと」を最優先とし、技術的な実現可能性を検証します。

- **アプローチ**: `ansible-core` の全コードを GraalPy の実行パスに含め、本物のコレクションが依存するライブラリをすべて参照可能な状態にします。
- **目標**: 複数の主要モジュール（`copy`, `file`, `stat` 等）が GraalPy 上で実際に動作することを確認します。
- **制約**:
    - 実行速度やメモリ使用量は度外視します。
    - Windows 環境での動作は必須とせず、まずは Linux/macOS での動作を確実にします。
- **主な作業**:
    - ビルド時に `pip` を使用して `ansible-core` および依存パッケージを取得する仕組みの導入。
    - GraalPy コンテキストへの `ansible-core` ライブラリパスの追加。
    - 依存する Python モジュールの不足解消。

## 3. フェーズ 2：ハイブリッド実装による Windows サポート

フェーズ 1 の成果をベースに、OS 互換性の向上を図ります。

- **アプローチ**: `ansible-core` をロードしつつ、Windows で動作しない、あるいは挙動が異なる一部のコンポーネントを、プロジェクト独自の Python 実装または Java ブリッジに差し替えます。
- **目標**: Windows 環境においても、主要なコレクションが正しく実行できる状態を目指します。
- **主な作業**:
    - Windows で動作を阻害する Python モジュールの特定とモンキーパッチ。
    - `OSHandler` と連携した、Windows 特有のパス処理やシェル実行の調整。

## 4. フェーズ 3：Ansible 本体のロード排除と最適化

最終段階として、パフォーマンスと安定性を極限まで高めます。

- **アプローチ**: コレクションの実行に本当に必要な最小限のライブラリ（`module_utils` のコア部分など）のみを抽出、あるいは再構成し、`ansible-core` 全体をロードする必要をなくします。
- **目標**:
    - 起動スピードの大幅な高速化。
    - 依存関係の最小化による、Native Image のビルド安定化とバイナリサイズの削減。
    - 外部の Python 環境に依存しない、スタンドアロンでの動作精度の向上。
- **主な作業**:
    - 必要なライブラリの依存関係解析とバンドル化。
    - 実行エンジンの最適化。
